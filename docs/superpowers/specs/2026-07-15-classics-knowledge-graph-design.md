# 古籍概念知识图谱拆解设计

日期：2026-07-15

## 1. 目标

将 `docs/wiki/` 中的 13 本古籍整理为可人工审阅、可关键词检索、可追溯原文、可扩展为图检索的知识库。

第一阶段不调用项目本地 LLM，也不批量生成白话。拆解结果由确定性解析程序产生，并由当前 Codex 会话对结构、概念和关系做质量校正。

交付物同时服务两类用途：

1. 问卦时从用户问题定位概念、规则和相关原典；
2. 人工浏览时可从书籍、主题或概念逐层查看，并回到原始出处。

## 2. 范围

### 2.1 输入

处理 `docs/wiki/` 当前全部 13 本书：

- 9 个纯文本文件（包括无扩展名的《柳庄相法》）；
- 1 个 EPUB《關聖帝君靈籤》；
- 3 个 PDF：《麻衣相法》第 1 卷、《三命通会》、《穷通宝鉴》。

原始文件只读，不在原位清洗或覆盖。

### 2.2 第一阶段包含

- 文本提取与格式归一化；
- 书籍、章节、段落的证据层拆分；
- 概念、方法、规则、案例的知识节点；
- 节点之间的显式关系；
- 每条知识到原典段落的引用；
- Markdown 知识库、总索引和机器可读图数据；
- 质量报告和待人工复核清单。

### 2.3 第一阶段不包含

- 调用项目配置的 DeepSeek、Mock 或其他本地/远程 LLM provider；
- 批量生成现代白话；
- 生成 embedding；
- 引入 Neo4j、ArangoDB 等图数据库；
- 修改问卦产品界面；
- 将未经复核的 OCR 文本直接写入生产数据库。

## 3. 设计原则

1. **证据优先**：任何概念定义和判断规则必须至少引用一个原文段落，推导性总结需明确标记为“归纳”。
2. **一页一主题**：概念、方法、规则各自独立成页，通过链接建立关系，避免在多个页面重复大段内容。
3. **原文不可变**：清洗只作用于中间文本；最终引用保留可辨认的原文，并记录来源位置。
4. **观点不强行合并**：不同书籍或流派结论不一致时，分别保存陈述并建立 `contradicts` 或 `differs_from` 关系。
5. **图谱轻量化**：Markdown 是权威的人类可读表示，`graph.json` 是其机器可读投影；第一阶段不依赖专用图数据库。
6. **确定性重跑**：稳定 ID、排序和输出路径必须可重复生成，方便后续增量更新和差异审阅。

## 4. 总体架构

```text
原始书籍
  │
  ├─ TXT/无扩展名：UTF-8 读取
  ├─ EPUB：按 spine 与标题层级提取
  └─ PDF：MinerU OCR/版面解析 → 页级质量标记
  │
  ▼
规范化中间文本
  │
  ▼
证据层：Book → Chapter → Passage
  │                         │
  │                         └─ 稳定 source_id
  ▼
知识层：Concept / Method / Rule / Case / Topic
  │
  ▼
关系层：知识节点之间的边 + 知识节点到原文证据的边
  │
  ├─ Markdown 页面与双向链接
  ├─ graph.json
  └─ quality-report.md
```

## 5. 输出目录

```text
knowledge_wiki/
├── index.md
├── log.md
├── domains/
│   ├── qian.md
│   ├── bazi.md
│   ├── physiognomy.md
│   ├── divination.md
│   └── cultivation.md
├── concepts/
│   ├── qian/
│   ├── bazi/
│   ├── divination/
│   ├── physiognomy/
│   └── cultivation/
├── methods/
├── rules/
├── cases/
├── intents/
├── sources/
│   ├── books/
│   └── passages/
├── graph.json
└── quality-report.md
```

`domains/` 是知识学科入口，`intents/` 是 APP 用户问题入口。`sources/books/` 保存每本书的目录、章节和段落索引。`sources/passages/` 可按书拆分，避免单文件过大。概念页面引用段落 ID，不复制整本书。

### 5.1 双轴分类

知识库不把“书籍学科”和“APP 殿堂”混成一棵目录树，而是采用两个可交叉的维度。

知识领域轴：

| domain | 中文名 | 主要来源 | 主要用途 |
|---|---|---|---|
| `qian` | 签筒灵签 | 《關聖帝君靈籤》 | 摇签、解签 |
| `bazi` | 八字命理 | 《渊海子平》《三命通会》《命理探源》《穷通宝鉴》 | 本命、八字、问卦 |
| `physiognomy` | 相法 | 《麻衣相法》《柳庄相法》《神相全编》 | 手相、面相报告 |
| `divination` | 易占 | 《梅花易数》《卜筮正宗》 | 问卦、卦象解释 |
| `cultivation` | 修身劝善 | 《了凡四训》《太上感应篇》《增广贤文》 | 建议、劝勉和兜底 |

用户意图轴：

| APP 入口 | intent | 优先知识域 |
|---|---|---|
| 文殊殿 | `exam` | 八字、灵签、修身 |
| 月老祠 | `love` | 八字、灵签、易占 |
| 财神殿 | `wealth`、`career` | 八字、灵签、易占 |
| 天机殿 | `natal` | 八字、相法 |
| 问卦 | 从问题自动识别 | 全部领域 |
| 摇签问事 | 用户所问主题 | 灵签优先，其他领域辅助 |
| 藏经阁 | `source` | 原文证据层 |

一个知识节点只保存一份，通过 `domain`、`intents` 和 `surfaces` 元数据映射到多个 APP 入口，禁止为不同殿堂复制同一概念页。

## 6. 数据模型

### 6.1 原文证据节点

每个段落包含：

```yaml
id: source:yuanhai-ziping:chapter-001:passage-0001
book_id: yuanhai-ziping
book_name: 渊海子平
chapter: 基础
text: 原文全文
page: null
sequence: 1
extraction: text
quality: verified
```

PDF 段落额外记录页码、OCR 引擎和置信状态。`quality` 取值为 `verified`、`review-needed` 或 `unusable`。

### 6.2 知识节点

统一字段：

```yaml
id: concept:bazi:ten-gods
type: concept
name: 十神
aliases: [十星]
domain: bazi
tags: [命理, 干支]
intents: [natal]
surfaces: [tianji, chat, library]
status: verified
```

节点类型：

- `concept`：术语、对象和基本概念；
- `method`：可执行的方法、步骤或计算过程；
- `rule`：带条件的判断规则；
- `case`：命例、占验、相例和反例；
- `topic`：财运、事业、婚姻、健康、功名等用户问题入口；
- `source`：书、章节和原文段落。

### 6.3 关系边

`graph.json` 中每条边至少包含：

```json
{
  "source": "concept:bazi:direct-wealth",
  "relation": "is_a",
  "target": "concept:bazi:ten-gods",
  "evidence": ["source:yuanhai-ziping:chapter-001:passage-0008"],
  "status": "verified"
}
```

允许的核心关系：

- `is_a`：属于；
- `part_of`：组成；
- `alias_of`：同名或异名；
- `generates` / `restrains`：生、克；
- `depends_on`：规则或方法依赖；
- `used_by`：概念被方法使用；
- `supports`：原典支持知识陈述；
- `example_of`：案例对应规则或概念；
- `contrasts_with`：相对或易混淆；
- `contradicts` / `differs_from`：来源间存在冲突或差异；
- `applies_to`：规则适用于主题或对象。

关系词表保持封闭；新增关系必须同时更新设计文档或关系定义文件。

## 7. 页面格式

### 7.1 概念页

```markdown
# 用神

> 一句话定义：判断卦或命局时选取的核心作用对象。

## 核心含义
## 使用范围
## 判断规则
## 容易混淆
## 不同来源的异同
## 相关概念
## 原典依据
```

页面顶部使用 YAML front matter 保存节点 ID、别名、领域和标签。正文链接只指向已经存在的知识页面。

### 7.2 规则页

规则必须显式区分：

- 前提条件；
- 判断过程；
- 结论；
- 例外或限制；
- 原典证据；
- 规则归属的书籍或流派。

不能把无条件格言误写为普遍可执行的预测规则。

### 7.3 来源页

来源页按章节列出稳定段落编号、原文、主题和术语标签。PDF 来源还显示页码与 OCR 质量标记。

## 8. 拆解流程

### 8.1 提取与归一化

1. 统一换行，去 BOM、零宽字符和排版空格；
2. 去除明确的下载说明、重复页眉页脚和纯页码；
3. 保留书名、作者、版本、来源 URL 等元信息；
4. EPUB 按 spine 顺序读取，保留 `h1` 至 `h3` 层级；
5. PDF 使用 MinerU `pipeline + ocr` 转换，保留 `content_list.json` 和坐标中间产物；
6. 《麻衣相法》竖排扫描页根据 MinerU 坐标按“列从右到左、列内从上到下”重排；
7. 空白页不当正文，OCR 无法可靠识别的页面标记 `review-needed`，不猜补文字。

### 8.2 章节与段落

1. 章节标题优先使用原始结构标记；
2. 纯文本使用卷、篇、章、节、签、卦以及编号模式识别；
3. 不再把所有六字以内短句自动视为标题；
4. 正文按 `。！？；` 聚合，目标约 120 字、软上限 220 字；
5. 签诗、歌诀、韵文和表格式口诀按原始语义单元保留，不强行拼成散文；
6. 每个段落生成基于书、章节和顺序的稳定 ID。

### 8.3 概念提取与合并

1. 从章节标题、术语词表和段落陈述中建立候选概念；
2. 归并繁简体、异体字和常见别名，但保留原文写法；
3. 同名异义概念按领域拆开，例如命理“用神”和卜筮“用神”分别建节点，再用关系说明联系；
4. 一个概念只有在存在明确定义、规则作用或多处引用时才独立成页；
5. 泛词和仅出现一次且无知识价值的词只保留为标签。

### 8.4 方法、规则和案例

- 有明确输入、步骤和输出的内容拆为 `method`；
- 可表达为“在条件 C 下，由依据 R 得出结论 O”的内容拆为 `rule`；
- 带人物、时间、命局、卦象、相貌或实际应验结果的内容拆为 `case`；
- 原文没有说明限制时，不自行补成绝对结论；必要的现代风险提示放在知识页说明区，不篡改原典。

### 8.5 APP 意图入口

意图页不是简单复制所有含关键词的段落，而是组织相关概念、方法、规则和证据。例如“财运”页连接正财、偏财、财星、财帛、田宅、求财签意等不同体系，并标明各自适用范围。

大类只承担路由和聚合职责：领域页按学科组织知识，意图页按用户问题组织知识。两者均链接到同一批概念节点、规则和原典，不产生重复内容。

## 9. 检索设计

第一阶段采用无需 embedding 的混合检索：

1. 用户问题匹配主题、概念名称和别名；
2. 对命中节点做一跳图关系扩展；
3. 根据规则适用范围过滤不相关体系；
4. 从关联证据段落中按标签和二元词重叠排序；
5. 返回知识结论、适用条件和原文出处。

图谱扩展默认只走一跳，避免概念扩散带来噪声。`contradicts` 关系命中时，结果必须并列呈现不同来源，不自动裁决真伪。

## 10. 与现有代码的关系

保留并复用 `apps/api/app/knowledge/` 中的归一化、句子聚合、主题分类和 wiki 导出思路，但需要在实施阶段扩展：

- 增加 EPUB 与 MinerU PDF 中间文本适配器；
- 修正短行标题误判；
- 扩展术语词表和领域分类；
- 增加稳定 ID；
- 增加概念、规则、方法、案例和关系边的导出；
- 让 wiki 可以先从文件产物生成，不要求数据库在线；
- 保留后续映射到 `Book` / `Passage` 以及新增图谱表的能力。

第一阶段不直接修改现有生产表结构。知识库通过质量验收后，再单独设计数据库迁移。

## 11. 错误处理与质量控制

### 11.1 提取错误

- 文件编码异常：记录文件名和字节偏移，停止该书处理，不用替换字符静默跳过；
- EPUB 章节缺失：按 spine 与目录交叉检查；
- PDF OCR 低质量：保留页图、页码和待复核标记，不生成引用结论；
- 无正文或段落过少：视为失败并写入质量报告。

### 11.2 知识错误

- 无来源的概念定义不得标记 `verified`；
- 同名概念跨领域不得自动合并；
- 冲突观点不得覆盖旧节点；
- 规则必须能回链至少一个证据段落；
- 删除或重命名节点时检查所有入边和出边，禁止产生悬空链接。

### 11.3 风险内容

相术、命理、疾病、婚育等原典陈述作为历史文献知识保存，但不包装成医学诊断或确定性现实承诺。检索输出应区分“古籍原文主张”和“现代事实”。

## 12. 验证与验收

### 12.1 自动检查

- 13 本输入均有处理结果或明确失败原因；
- 所有知识节点 ID 唯一；
- 所有关系端点存在；
- 所有已验证规则至少有一个证据 ID；
- 所有 Markdown 内链可解析；
- `graph.json` 可被标准 JSON 解析；
- 同一输入重复运行时，稳定 ID 和排序不变化；
- 统计清洗前后字符数，异常丢失超过阈值时阻止交付；
- PDF 页数与 OCR 页级记录数一致，空白页需显式标记。

### 12.2 人工抽样

每本书至少抽查：

- 第一章、中间章节和末章各一组段落；
- 两个概念节点；
- 一个判断规则；
- 一个跨页或复杂排版位置；
- PDF 每种 OCR 质量等级至少一页。

### 12.3 验收标准

- 可以从概念页回到准确的书名、章节和原文段落；
- 可以从原文段落找到其涉及的概念和规则；
- 不存在明显章节错位、句中截断或来源混淆；
- 相同概念跨书聚合，但不同体系和冲突观点仍可区分；
- 在不运行本地 LLM 的情况下可完整浏览、检索和重建产物。

## 13. 实施顺序

1. 建立输出规范、稳定 ID 和质量检查器；
2. 处理 9 本纯文本并修正章节规则；
3. 处理 EPUB《關聖帝君靈籤》；
4. 使用 MinerU 中文 OCR，逐本处理 3 个 PDF；
5. 生成证据层知识库；
6. 建立概念、方法、规则、案例和主题节点；
7. 生成关系边、索引和图数据；
8. 跑自动校验与人工抽样，形成质量报告；
9. 用户复核后再设计数据库入库与白话补充阶段。
