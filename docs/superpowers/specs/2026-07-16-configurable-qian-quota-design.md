# 求签次数可配置设计

日期：2026-07-16  
状态：已获用户方向确认，待书面复核

## 1. 目标

将当前写死在业务常量中的求签每日次数改为部署配置，并在当前阶段关闭求签次数限制。问卦额度、长明灯权益和其他业务限额不受影响。

## 2. 方案比较

1. **环境变量配置（采用）**：增加 `QIAN_DAILY_LIMIT`，部署时即可切换，无需数据库或管理后台；适合当前阶段。
2. **继续使用代码常量**：改动最小，但每次调整都要改代码并重新发布，不满足“可配置”。
3. **数据库加管理后台**：可运行时修改和审计，但需要权限、缓存和管理界面，当前需求下成本过高。

## 3. 配置语义

- `QIAN_DAILY_LIMIT=0`：求签无限制，作为当前默认值。
- `QIAN_DAILY_LIMIT=N`，其中 `N` 为正整数：所有用户每日最多求签 `N` 次。
- 负数或非整数：应用启动时配置校验失败，避免带着含糊配置运行。
- 该配置只覆盖 `qian`；`chat` 仍沿用现有免费用户与长明灯用户额度。

## 4. 后端设计

在 `Settings` 增加非负整数 `qian_daily_limit`。`quota.limit_for()` 对 `qian` 优先返回该配置；其他额度继续读取 `FREE_QUOTA` / `LAMP_QUOTA`。

`QuotaOut` 增加 `unlimited: bool = False`，避免用超大数字或负数暗示无限：

- 无限模式下，`get_quota()` 返回 `limit=0`、`remaining=0`、`unlimited=true`；
- 无限模式下，`consume()` 直接返回，不写入 `daily_quota`，也不会抛出 429；
- 正整数模式下，继续使用现有 PostgreSQL upsert 原子计数和超限回滚逻辑。

这样求签服务仍统一调用 `quota.consume()`，限额开关只存在于 quota 模块，不侵入抽签逻辑。

## 5. 前端设计

前端 `Quota` 类型增加 `unlimited`。签室顶部根据 API 契约展示：

- `unlimited=true`：显示“今日求签 · 不限”；
- 否则：显示“今日尚可求签 N 次”。

前端不自行判断 `limit === 0`，以免将未来可能的“暂停服务”语义误认成无限。

## 6. 错误与兼容

- `unlimited` 在响应模型中有默认值，兼容内部仍按旧结构构造 `QuotaOut` 的调用。
- 已存在的 `daily_quota` 历史记录保留；无限模式不继续累计，恢复正整数限制后按当天已有计数继续计算。
- 配置变更通过重启 API 生效，不引入运行时热更新。

## 7. 验证

- 配置默认值为 `0`，负数校验失败；
- 无限模式下连续求签不会写额度计数、不会返回 429；
- 正整数模式仍能原子计数并在超限时返回 429；
- `/api/qian/quota` 正确返回 `unlimited`；
- 桌面端显示“不限”，现有动画、解读和收藏流程不回归；
- 后端 pytest、Ruff 与前端生产构建通过。

## 8. 范围外

本次不增加管理后台、运行时动态配置、按用户分组限额或新的付费策略。
