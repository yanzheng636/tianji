# 天机寺桌面 Web 产品设计规格

日期：2026-07-15

## 1. 项目目标

在保留现有 `apps/web` 移动端风格版本的前提下，新建独立的桌面网页应用 `apps/web-desktop`。新应用复用现有后端 API 与业务闭环，但重新设计信息架构、视觉语言和交互方式，成为真正面向终端用户的桌面 Web 产品。

产品定位仍为“传统文化娱乐 + 心理疗愈”。它不预测命运、不承诺改命。桌面版的体验目标是让用户感到自己进入了一座安定、可信、可以探索的线上寺院，而不是进入管理后台、功能导航站或被放大的手机应用。桌面版不使用手机号认证，首次访问时自动建立匿名访客会话。

## 2. 设计原则

1. **产品，而非后台**：不采用常驻左侧菜单、仪表盘卡片矩阵和数据后台式表格。
2. **场景承载功能**：首页寺院建筑必须对应真实功能和状态，不做无意义装饰。
3. **仪式与效率并存**：首页负责氛围和探索，功能内页负责清晰、快速地完成任务。
4. **克制的东方性**：使用建筑秩序、材料、光线和排版表达传统文化，避免书法字体滥用、素材拼贴和仙侠游戏化。
5. **可信与合规**：AI 解读明确展示古籍来源、娱乐性质与必要免责声明。
6. **旧版零影响**：不修改 `apps/web` 的产品结构；两个前端独立构建、独立发布。

## 3. 核心概念：一日入山

采用“游寺叙事”路线。首页以寺院中轴线作为浏览路径，空间场景建立情绪，现代 Web 交互负责完成任务。

主场景处于晨昏交界：远山与屋瓦是冷青色，院内第一批灯火已亮。冷色环境建立安静感，暖色灯火标记可交互地点和用户行为。整体神秘但不压抑，文化感强但不仿古。

标志性交互是“灯火”。用户悬停建筑、完成上香、写下愿望或开始问卦时，对应位置产生克制的灯光反馈。登录后，首页可呈现用户当前的香火、愿望、签文或供灯状态。灯火是反馈和状态表达，不设计成积分或游戏收集机制。

## 4. 信息架构

### 4.1 一级导航

桌面顶栏包含：

- 天机寺品牌标识：返回寺院首页。
- 游寺：进入完整寺院场景。
- 问卦：直接进入 AI 问卦。
- 藏经阁：进入古籍浏览与阅读。
- 许愿池：浏览和发布愿望。
- 灯火状态：显示正在燃烧的香或长明灯权益。
- 个人入口：命盘、历史记录与访客资料设置。

一级导航不承载全部功能。摇签、上香、供灯等仪式型功能通过寺院场景与相关状态入口进入。

### 4.2 场景与功能映射

| 场景 | 主要功能 | 首页状态摘要 |
|---|---|---|
| 山门 | 新用户引导、今日寺语、开始游寺 | 今日寺语 |
| 签房 | 摇签、签文解读、历史签文 | 今日剩余次数、最近一签 |
| 香火殿 | 上香、燃香计时、供灯 | 当前燃香、灯火状态 |
| 问天殿 | AI 问卦、继续历史对话 | 最近会话、可用追问 |
| 许愿池 | 写愿、公共愿望、祈愿互动 | 最近愿望、公共动态 |
| 藏经阁 | 古籍目录、阅读、检索与引用 | 今日推荐、最近阅读 |
| 我的居所 | 八字命盘、个人资料、权益与记录 | 命盘摘要、身份状态 |

### 4.3 路由结构

- `/`：寺院首页
- `/temple`：完整游寺场景
- `/qian`：签房
- `/incense`：香火殿
- `/chat`：问天殿
- `/wishes`：许愿池
- `/library`：藏经阁
- `/library/:bookId`：古籍阅读
- `/profile`：我的居所
- `/profile/chart`：完整命盘
- `/lamp`：供灯与权益

浏览器前进、后退、刷新和深链接均需正常工作。应用不得只用全局状态模拟路由。

## 5. 首页设计

### 5.1 页面构成

首页分为四层：

1. 顶部产品导航。
2. 占据首屏主体的国风数字寺院插画。
3. 建筑热点与状态浮层。
4. 场景下方的“今日与继续”内容带。

桌面首屏应在 1440×900 下完整看到寺院主体、主要建筑热点和下方内容带的开头。不得依赖滚动后才知道产品能做什么。

### 5.2 建筑热点

建筑默认不显示大面积标签。鼠标移入时：

- 对应建筑局部亮灯；
- 雾气或遮罩轻微退开；
- 出现殿名、一句状态和一个明确动作；
- 点击进入功能页；
- 键盘聚焦产生等价反馈；
- `prefers-reduced-motion` 下取消位移动画，仅保留透明度和描边变化。

热点必须有合理的可点击区域、可访问名称与明显焦点，不依赖精确点击建筑细节。

### 5.3 今日与继续

场景底部使用一条横向内容带，最多展示四项与当前用户最相关的内容：今日寺语、最近签文、未完成的愿、继续上次问卦。匿名会话尚未建立时展示今日寺语和功能导览；会话建立后再加载个性化数据，不伪造状态。

## 6. 核心功能内页

### 6.1 AI 问卦

采用“书签目录 + 对谈主卷 + 古籍旁证”的宽屏三段式布局：

- 左侧：历史问卦书签，可折叠；
- 中间：主要对话、流式回答、推荐追问和输入区；
- 右侧：古籍出处、原文片段与解读依据，可折叠。

视觉上三部分属于同一张展开案卷，不做三个彼此割裂的后台卡片。中间栏保持主要视觉权重。小于 1180px 时右栏收为抽屉；小于 900px 时左栏也收为抽屉。

流式异常需保留已生成内容，并提供“继续生成”或“重新提问”；引用加载失败不能阻断主回答，但必须明确显示出处暂不可用。

### 6.2 签房

页面由主题选择、摇签仪式、签文结果和古籍解读组成。摇签动画集中在一次明确的仪式中，不散布无意义粒子效果。结果页充分利用横向空间：左侧展示签纸，右侧展示签级、释义、提醒与出处。历史签文以时间轴或签夹形式展示。

### 6.3 香火殿与供灯

主区域展示当前燃香状态、剩余时间和所寄之愿；次区域展示可选香型与历史记录。服务端 `endsAt` 是唯一计时依据，前端恢复页面时重新校准。供灯是独立的权益说明与购买流程，强调实际权益，不使用“改运”“消灾”等承诺性文案。

### 6.4 许愿池

公共愿望呈现在院墙木牌式信息流中，但文本排版和筛选操作遵循现代网页可读性。写愿入口始终明确。内容审核中、审核未通过、提交失败均提供直接说明和下一步动作。公共区不展示手机号等个人信息。

### 6.5 藏经阁

目录页使用“书架索引 + 今日推荐 + 最近阅读”结构。阅读页使用适合长文的居中正文栏，并提供章节目录、检索、引用定位和返回问卦的操作。古籍原文、现代释义和 AI 补充必须在视觉上明确区分。

### 6.6 我的居所与命盘

个人页不做账户后台。首屏展示用户称谓、当前灯火和命盘摘要；完整命盘进入独立页面。命盘使用桌面宽度呈现四柱、五行分布、基础解读和出生信息编辑。信息缺失时提供补全引导，不显示空白仪表盘。

## 7. 视觉系统

### 7.1 色彩

- 远山青 `#526866`：环境、次级区域。
- 殿宇墨 `#182321`：主文字、深色结构与夜色背景。
- 香火金 `#C69245`：重点、选择态和文化性细节。
- 灯芯橙 `#E27A3F`：主动行为、灯火与进度反馈。
- 月瓷白 `#E9E6DA`：阅读背景和浅色正文区。
- 木漆红 `#7B3028`：危险以外的品牌强调、印记与殿名。

色彩对比需满足 WCAG AA。金色不直接用于小字号正文；灯芯橙只用于高价值动作与状态，不成为全站按钮默认色。

### 7.2 字体与排版

- 展示标题：选用具有碑刻骨架、适合屏显的中文宋体或宋黑混合字体。
- 正文：现代屏显宋体，保障长文阅读。
- 界面与数字：中性窄体无衬线，用于导航、时间、状态和表单。
- 书法：只用于寺名印记与签文题头，且必须有本地或合法授权字体资源。

正文默认 16px，长文行高不低于 1.8，正文栏控制在约 68–76 个中文字符宽度。页面标题不靠夸张字号制造层级，而由空间、材质和结构共同完成。

### 7.3 材质与形状

界面取材自旧木、石阶、窗棂、瓷白纸面和灯罩，但以颜色、边界和轻微纹理表达，不使用拟真贴图覆盖所有容器。圆角克制：正文纸面和弹层可使用 12–16px，结构区与导航更接近 4–8px。

### 7.4 动效

仅保留三类动效：

1. 首页建筑亮灯与薄雾变化；
2. 摇签和上香的单次仪式动画；
3. 页面进入时的轻微层次展开。

全站尊重减少动态效果设置。动画不得延迟主操作，不使用持续漂浮、鼠标尾迹和大面积粒子背景。

## 8. 响应式策略

新应用以桌面浏览为核心，支持以下区间：

- `>= 1440px`：完整寺院构图与多栏内页；
- `1180–1439px`：压缩场景边距，内页可折叠次级栏；
- `900–1179px`：保留顶部导航，寺院热点重新排布，复杂侧栏进入抽屉；
- `< 900px`：提供可用的紧凑版作为降级体验，但不替代现有 `apps/web` 移动版。

桌面版不得通过固定手机宽度容器居中实现。首页插画需要定义安全裁切区，保证主要建筑在常见宽高比下不被切断。

## 9. 前端架构与组件边界

应用使用 React、TypeScript、Vite，并复用现有 API 契约；路由层采用 React Router。状态分为三类：

- 服务端状态：用户、签文、愿望、香火、会话、古籍，由请求缓存层管理；
- 会话 UI 状态：抽屉、热点、当前动效阶段，保持在页面或组件内；
- 跨页面客户端状态：认证信息、全局通知、用户轻量偏好，放在精简的全局 store。

核心组件边界：

- `AppShell`：顶栏、路由出口、全局反馈；
- `TempleScene`：场景渲染与安全裁切；
- `TempleHotspot`：无障碍热点、状态摘要与交互反馈；
- `ContinueStrip`：今日与继续内容；
- `ReadingLayout`：藏经阁长文布局；
- `ConsultationLayout`：问卦多栏布局；
- `RitualStage`：摇签与上香的可控仪式阶段；
- `SourceCitation`：统一的古籍引用展示；
- `GuestSessionBoundary`：自动建立并恢复匿名访客会话；
- `EmptyState`、`ErrorState`、`LoadingState`：统一状态组件。

不得继续以大量 JSX 内联样式组织页面。颜色、字体、间距、层级和动效建立为 CSS 变量与有限的布局原语；业务组件保持可读和可测试。

## 10. 数据流与 API 复用

新前端继续调用现有 `/api/*` 契约，不修改服务端业务权威原则：

- 首次访问自动创建匿名访客并保存认证令牌，后续访问恢复会话；
- 摇签由服务端返回结果与配额；
- 上香以服务端结束时间计算状态；
- 许愿先经过服务端审核；
- 问卦使用 SSE 流式响应并展示 RAG 引用；
- 命盘由服务端计算；
- 支付按现有订单状态轮询或回调后的查询结果更新。

桌面端可新增聚合型首页请求以减少首屏瀑布流，但必须由后端明确提供，不在前端拼接伪状态。首版若不改后端，首页各状态模块应独立加载和失败，不能因一个接口失败导致整个寺院场景不可用。

## 11. 异常、空状态与权限

- 匿名会话创建中：允许游寺、浏览公开内容；执行受限动作时等待会话就绪后继续原动作。创建失败时提供重试，不要求手机号。
- 网络异常：首页场景与静态入口仍可浏览；动态状态标记为暂不可用。
- 空历史：明确引导首次摇签、问卦或写愿，不展示空白列表。
- 配额用尽：解释恢复时间和可选权益，不使用焦虑营销。
- 审核失败：说明内容未公开及修改方向。
- 支付等待：展示可恢复的订单状态，刷新页面后仍能继续查询。
- AI 输出：固定位置展示娱乐与文化参考声明；高风险主题提供合适的安全提示。

## 12. 可访问性与性能

- 所有建筑热点可通过键盘访问，拥有可读名称和明显焦点。
- 场景插画提供文本等价导航，不能成为唯一入口。
- 表单拥有真实标签、错误关联和提交状态。
- 色彩不是状态的唯一表达方式。
- 首页核心插画按设备尺寸提供多规格资源，优先加载可见区域。
- 首屏不加载所有建筑内页素材；路由级代码分割。
- 字体子集化并设置合理回退，避免字体阻塞页面。
- 目标：桌面环境 LCP 不高于 2.5 秒，CLS 不高于 0.1，常用交互 INP 不高于 200ms。

## 13. 测试与验收

### 13.1 功能验收

- 新旧应用可独立启动、构建和部署。
- 桌面版完整覆盖现有业务功能与认证闭环。
- 所有路由可刷新、深链接和浏览器回退。
- 匿名会话建立后可继续此前触发的受限动作。
- SSE 中断、计时恢复、审核失败和支付等待均有可恢复体验。

### 13.2 视觉验收

- 1440×900 首屏能识别寺院主体、核心建筑和产品用途。
- 首页不呈现后台侧栏或卡片仪表盘观感。
- 问卦、藏经阁、命盘三类页面形成一致但各自适配内容的桌面布局。
- 900、1180、1440、1920px 宽度无水平溢出和关键内容遮挡。
- 减少动态效果模式下功能完整。

### 13.3 自动化与人工验证

- TypeScript 类型检查和生产构建；
- 路由与核心状态组件单元测试；
- 登录、摇签、上香、许愿、问卦、命盘、支付关键流程集成测试；
- 键盘操作与基础无障碍扫描；
- 主流桌面浏览器截图回归；
- 真实后端或契约一致的测试服务完成端到端验收。

## 14. 首版范围与非目标

首版包含：独立应用骨架、首页寺院场景、全部现有业务页面的桌面重构、匿名访客会话、响应式降级、基础动效、可访问性与错误状态。

首版不包含：3D 自由漫游、多人实时同屏、游戏化成长体系、全套日夜实时天气、原生移动端替代、后端业务重写。首页可先采用分层 2D 插画和局部动效，为未来扩展保留接口。

## 15. 实施顺序

1. 建立独立应用、路由、主题令牌和 API 适配。
2. 完成静态首页场景、热点系统和顶栏。
3. 完成问卦、藏经阁、命盘三个桌面布局样板。
4. 迁移签房、香火殿、许愿池和供灯流程。
5. 接入首页个性化状态、匿名会话续接与异常恢复。
6. 补齐动效、响应式、无障碍、性能和回归测试。
