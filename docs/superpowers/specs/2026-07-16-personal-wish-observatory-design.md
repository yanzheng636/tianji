# 个人愿星图设计说明

## 目标

把桌面端“愿池”从公共愿望展示页改成仅当前用户可见的个人愿望空间。页面需要同时完成三件事：让许愿动作有仪式感、让愿望历史形成持续生长的星图、让用户能清楚管理尚未还愿的愿望。

页面不展示他人愿望，也不出现“审核中、审核通过、审核拒绝”等后台状态。

## 最终体验

### 首屏：个人愿星图

- 标题使用“念有所向，微光自来”，保留山问现有墨绿、骨白、铜金、暗朱色视觉语言。
- 每条个人愿望对应一颗愿星。
- `active` 愿望使用铜金亮星；`fulfilled` 愿望保留在星图中，但降为柔和灰绿余光。
- 愿望增加时，星星数量随之增加；星星位置由愿望 ID 稳定生成，刷新页面不重新跳位。
- 星图只在聚焦、悬停或选择时显示愿望正文，避免文字同时铺开造成杂乱。
- 首屏明确标注“这片星图仅自己可见”。

### 许愿仪式

- 点击“写下一颗愿星”后，在同一片墨夜背景中升起输入区域，不使用独立浅色卡片。
- 用户提交后，文字凝聚成一束微光并进入星图。
- 请求成功后立即显示“愿望已记下”，并把新愿望插入星图与下方列表。
- 界面不显示后台审核过程；审核字段只作为服务端防御能力保留。
- 提交失败时保留原输入，只提示可操作的网络或服务错误。

### 下方：尚未还愿

- 只列出当前用户 `status === "active"` 的愿望，按创建时间倒序排列。
- 每行展示愿望正文、创建日期和“已经实现，去还愿”动作。
- 还愿成功后，该愿望立即从列表移除；对应星星仍保留，但切换为余光状态。
- 没有未还愿望时，展示“此刻没有尚未还的愿，写下一颗新的愿星”的空状态。

## 隐私与数据语义

为了让“仅自己可见”成为真实行为，而不是只隐藏桌面端 UI：

- `GET /api/wishes` 的 `total` 改为当前用户的个人愿望总数。
- `mine` 只返回当前用户自己的愿望，包含进行中和已还愿记录。
- `floating` 不再返回公共愿望，固定为空数组；现有字段暂时保留以避免破坏客户端契约。
- 未建立用户会话时返回 `total = 0`、`floating = []`、`mine = []`。
- 许愿内容仍会落库；审核结果不影响愿望在本人星图和列表中的可见性。
- 桌面端与现有移动 Web 都不再读取或展示他人的愿望。

服务端个人记录保留合理上限以防止异常大响应；本次使用最近 200 条个人愿望形成星图，`total` 始终保留完整计数。超过可视上限时，首屏继续通过背景微星密度表达累积数量，不再增加可交互标签。

## 代码边界

### 桌面端页面

`apps/web-desktop/src/pages/Wishes.tsx`

- 负责加载个人愿望、许愿、还愿和本地乐观更新。
- 将星图位置生成拆成无副作用的小函数，输入为愿望 ID，输出为稳定的坐标、大小和动效延迟。
- 只渲染个人数据，不再读取 `floating`。
- 保留现有 API 调用，不引入新的前端依赖。

### 样式

`apps/web-desktop/src/styles.css`

- 替换现有 `wish-ledger-*` 视觉块，新增个人星图、输入仪式、未还愿列表和空状态样式。
- 桌面端首屏保持完整沉浸空间；小于 980px 时调整为上下结构；小于 700px 时压缩星图并把列表动作改为整行按钮。
- 提供清晰的键盘焦点，遵守 `prefers-reduced-motion`，关闭星星漂移和提交飞行动画。

### 现有移动 Web

`apps/web/src/screens/Wish.tsx`

- 同步改为只读取 `mine`，避免服务端隐私语义改变后仍保留公共池文案。
- 许愿后只提示“愿望已记下”，个人列表不显示审核状态。
- 保留现有移动端结构和交互，不在本次重做其视觉系统。

### 服务端

`apps/api/app/services/wish.py`

- 将愿池查询从公共抽样改为当前用户私有查询。
- `total` 统计当前用户全部愿望；`floating` 固定为空；`mine` 返回最近 200 条个人记录。
- 不改变创建和还愿接口路径。

`apps/api/app/schemas.py`

- 保留 `WishPoolOut` 结构，更新注释，避免把 `moderation` 描述为前端公开状态。

## 状态与错误处理

- 加载中：星图区域使用克制的呼吸光点，不伪造愿望数量。
- 加载失败：保留页面骨架，展示“愿星暂未显现，重新查看”操作。
- 提交中：禁用重复提交，按钮文案为“正在点亮”。
- 提交失败：输入内容不清空。
- 还愿中：只禁用当前愿望的按钮。
- 还愿失败：愿望留在列表中，并显示行内错误。

## 验证

### 服务端测试

- 已登录用户只能得到自己的愿望。
- `total` 包含自己的进行中和已还愿记录，不受审核状态影响。
- `floating` 始终为空。
- 未登录请求不泄露任何愿望。
- 创建后愿望立即出现在本人结果中。
- 还愿后状态切换为 `fulfilled`，记录仍保留。

### 前端验证

- 空愿望、单个愿望、多个愿望、混合进行中/已还愿状态。
- 新建愿望后星图和列表立即增加。
- 还愿后列表移除、星星保留并变暗。
- 网络错误不会丢失输入或错误移除愿望。
- 1440px、980px、700px 和约 390px 宽度下无横向溢出。
- 键盘可完成许愿和还愿；减少动态效果设置生效。

## 不在本次范围

- 公开愿望、愿望社交互动、点赞或评论。
- 单独的已还愿列表或详情页；已还愿历史只在星图中保留。
- 删除愿望、编辑愿望或批量还愿。
- 新增通知、分享或导出能力。
