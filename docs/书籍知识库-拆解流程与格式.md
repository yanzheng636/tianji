# 书籍知识库 · 拆解流程与格式

> 把古籍原文拆解、分类、入库，供「问卦」引经据典、可溯源。
> 风格参考 CowAgent 的 Knowledge：**结构化 wiki（索引 + 分类）+ 关键词检索**，向量作为可选增强。
> 对应代码：`apps/api/app/knowledge/`（`chunker.py` / `topics.py` / `ingest.py` / `wiki.py`）。

---

## 0. 总览

```
原文 .txt/.md
   │
   ▼  ① 归一化           normalize：统一换行、去全角空格/零宽字符
   ▼  ② 章节切分         按「标题行」切成若干章节（卷/篇/章/第…签/markdown #）
   ▼  ③ 段落聚合         章节正文按句末标点聚成「不破句」的段，目标 ~120 字
   ▼  ④ 主题 + 标签       每段自动打 topic（考/财/姻缘/事业/健康/通用）+ 术语 tags
   ▼  ⑤ (可选) 白话       LLM 逐段生成现代白话（需真实模型，mock 不开）
   ▼  ⑥ 落库             写入 Book / Passage 表（幂等，按 slug 覆盖）
   ▼  ⑦ 导出 wiki        生成 index.md + books/<slug>.md + topics/<topic>.md
   │
   ▼
问卦检索：主题过滤 + 标签命中 + 二元词(bigram)重叠打分 → 取 top-K 作为引用
```

一句话：**原文进，一段段带「出处 / 原文 / 白话 / 主题 / 标签」的结构化知识出**，既能被机器检索，也能被人翻阅。

---

## 1. 输入：你给我什么

- **格式**：纯文本 `.txt` 或 `.md`，UTF-8。
- **内容**：古籍**原文**即可（清及以前，公有领域）。
- **来源建议**：`ctext.org`（中国哲学书电子化计划）、`zh.wikisource.org`（维基文库）。
- ⚠️ **不要**给现代出版社的「白话译注 / 校注」版——译文与注释有版权。白话由 LLM 生成或人工补，不抄他人。

对排版无洁癖要求：有没有章节标题、有没有空行都能处理，下面的规则会尽力识别结构。

---

## 2. ① 归一化（normalize）

进入拆解前先把文本洗干净，避免格式噪声影响切分：

- `\r\n` / `\r` → `\n`
- 全角空格 `　` → 半角空格
- 去除零宽字符（BOM 等）

---

## 3. ② 章节切分（识别「标题行」）

逐行扫描，遇到「标题行」就开一个新章节，其余行归入当前章节正文。

**判定为标题行的情况**（`topics.looks_like_heading` + markdown 规则）：

| 类型 | 例子 |
|---|---|
| Markdown 标题 | `## 卷二 掌纹篇` |
| 卷/篇/章/节/签/卦 | `卷一`、`第七签`、`第三章`、`乾卦第一` |
| 中文序号条目 | `一、`、`二．` |
| `X 篇`/`X 章` 短行 | `掌纹篇`、`官禄篇` |
| 极短且无句末标点的行 | `气色`（≤6 字、不含 `。！？；`）|

> 设计取舍：文言文里标题多为短行、无句末标点；正文句子短、标点密集。用「**短 + 无句末标点 + 命中关键词模式**」来区分标题与正文，绝大多数古籍排版都能对上。识别不到标题时，整段归入默认章节名（默认「正文」，可用 `--chapter` 指定）。

产物：`[(章节名, [正文行, ...]), ...]`

---

## 4. ③ 段落聚合（不破句 + 目标长度）

章节正文往往是一大段连续文字，需要切成检索友好的「段」。规则：

1. 先按**句末标点** `。！？；` 把正文切成句子（切分点保留在句尾，不丢标点）。
2. 再把句子**从前往后聚合**成段，直到长度达到目标：
   - `target = 120` 字：达到即收一段；
   - `hard_max = 220` 字：硬上限，**绝不在句子中间断开**（宁可略超目标也凑完整句）；
   - `min_len = 8` 字：太短的碎片丢弃。

> 为什么按句聚合而不是定长切：文言一句往往自成语义单元（「天行健，君子以自强不息」），从中间切开会毁掉可引用性。所以**永远整句进出**，长度只是聚合的停止信号。

参数都可调（`chunk_text(target=, hard_max=, min_len=)`），文言偏短句，故默认取 120/220。

---

## 5. ④ 主题分类 + 标签抽取（`topics.py`）

对每一段做两件事，都是**关键词规则**（古籍术语固定，命中率高，零成本、可解释）：

**主题 `topic`**（六选一，取命中最多者，全不中为 `general`）：

| topic | 触发词（节选） |
|---|---|
| `exam` 升学考试 | 文昌、科甲、功名、登科、状元 |
| `career` 事业官禄 | 官禄、事业、仕、禄、贵人、升迁 |
| `wealth` 财运财帛 | 财、富、田宅、帛、库、利 |
| `love` 姻缘感情 | 姻缘、夫妻、婚、桃花、红鸾、夫宫/妻宫 |
| `health` 健康气色 | 疾、病、寿、气色、精神、命门 |
| `general` 通用 | 以上都不命中 |

**标签 `tags`**（术语级，用于检索加权，最多 6 个）：
从固定术语表里挑命中的词，如 `手相 / 掌纹 / 面相 / 气色 / 八字 / 五行 / 六爻 / 文昌 / 官禄 / 财帛 / 姻缘 / 印堂 / 明堂 / 生命线 / 智慧线 / 感情线 / 劝善 / 改过 …`。

> 主题决定「这段归哪个殿」，标签决定「关键词检索时的命中权重」。两者都可后续接 LLM 做更细的分类，但规则版已足够可用。

---

## 6. ⑤（可选）LLM 生成白话

默认**不生成**白话（`plain=""`），前端可只显示原文。

加 `--distill` 时，逐段请 LLM：

> 「把这句文言，用一句不超过 40 字的现代白话说清楚，只输出白话本身。」

- 需真实模型（`LLM_PROVIDER=deepseek`），mock 下产出的是规则话术而非翻译，故 mock 别开。
- 白话是**生成**的，不抄版权译注 —— 这也是「你只给原文」的原因。

---

## 7. ⑥ 落库：数据结构

拆出来的段落写入两张表（`app/models.py`），**按 `slug` 幂等**：重复入库先删旧的同名书再写。

**Book（一本书）**
| 字段 | 说明 |
|---|---|
| `slug` | 唯一标识，如 `mayi` |
| `name` | 书名，如「麻衣相法」|
| `char` | 书名首字，UI 书脊显示 |
| `meta` | 副标题，如「相法 · 三卷 · 宋」|
| `sort` | 排序 |

**Passage（一段）**
| 字段 | 说明 |
|---|---|
| `book_id` | 所属书 |
| `chapter` | 出处章节，如「卷二 · 掌纹篇」（溯源用）|
| `text` | **原文**（整句，可引用）|
| `plain` | 白话（可空）|
| `topic` | 主题（六类之一）|
| `tags` | 术语标签数组（检索加权）|
| `embedding` | 向量，**当前留空**；接 pgvector 时在此填，检索层自动切换 |
| `sort` | 段序 |

> `embedding` 列现在是 `null`，检索走关键词。这一列的存在让「日后接向量」变成**填数据 + 换打分函数**，不用改表结构、不用重构。

## 8. ⑦ 导出 wiki（CowAgent 式结构）

入库后可 `--wiki` 导出一份人可读、可版本管理的 Markdown 知识库：

```
knowledge_wiki/
├── index.md                 总入口：书目清单 + 主题索引链接
├── books/
│   ├── mayi.md              《麻衣相法》：章节 → 段落（原文/白话/标签）
│   └── liaofan.md
└── topics/
    ├── wealth.md            「财运财帛」主题下所有段落（跨书聚合，带出处）
    ├── exam.md
    └── ...
```

- `index.md`：书目 + 各主题段落数，作为「索引式检索」的入口。
- `books/<slug>.md`：按章节组织，每段列出原文（＋标签）＋白话。
- `topics/<topic>.md`：把同主题段落跨书聚到一起，每条带「——《书名 · 章节》」出处。

这对应 CowAgent 的三种取用方式：**索引查（index.md）→ 语义/关键词搜 → 直接读文件（topics/books md）**。

---

## 9. 检索：问卦时怎么用这些段

问卦时（`services/scripture.retrieve`）按用户问题打分，取最相关的 1~K 段作为**引用卡片**回给大师：

**关键词打分（当前默认）** = 主题命中 + 标签命中 + 二元词重叠：

| 信号 | 权重 | 说明 |
|---|---|---|
| 主题相符（问题主题 == 段主题）| `+2.0` | 问财运优先取 wealth 段 |
| 标签命中（段的 tag 出现在问题里）| 每个 `+1.2` | 术语级强信号，如问题含「文昌」|
| 二元词(bigram)重叠 | 每个 `×0.6` | 「文昌」比零散「文」「昌」更可信，比单字匹配准 |

> 用 **bigram** 而非单字：中文单字匹配噪声大（「不」「之」到处都是），相邻两字才承载语义。这对术语固定的古籍尤其有效。

**向量（未来增强）**：一旦 `embedding` 有值且配了 embedding provider，检索层自动改用「查询向量 × 段向量」余弦相似度 + 主题微调，无缝替换关键词打分。当前刻意先不接（体量小、关键词够用）。

无命中时兜底返回一条通用劝勉，保证大师总能「引一句」。

## 10. CLI 用法

```bash
cd apps/api

# 基本入库（关键词检索，不生成白话）
python -m app.knowledge.ingest 麻衣相法.txt \
    --slug mayi --name 麻衣相法 --meta "相法 · 三卷 · 宋"

# 让 LLM 逐段生成白话（需真实模型）
python -m app.knowledge.ingest 了凡四训.txt \
    --slug liaofan --name 了凡四训 --meta "劝世 · 明 · 袁黄" --distill

# 入库后同时导出 wiki
python -m app.knowledge.ingest 周易.txt \
    --slug zhouyi --name 周易 --meta "易占 · 先秦" --wiki
```

参数：`--slug`（必填唯一标识）、`--name`（书名）、`--meta`（副标题）、`--chapter`（无标题时的默认章节名）、`--distill`（生成白话）、`--wiki`（导出 wiki）。按 slug 幂等，可反复重跑。

### 10.1 全量古籍概念图谱（当前方案）

`docs/wiki/` 全部书籍使用新的确定性构建器统一处理；PDF 固定通过 MinerU `pipeline + ocr` 转换，过程不调用项目的 LLM provider：

```bash
cd apps/api

python -m app.knowledge.graph_builder \
    --input-dir ../../docs/wiki \
    --output-dir ../../knowledge_wiki \
    --mineru-cache ../../tmp/mineru \
    --mineru-device mps \
    --mineru-source modelscope
```

输出以 `knowledge_wiki/index.md` 为总入口，同时生成五大知识领域、APP 意图页、概念/方法/规则/案例页、灵签签号聚合页、可追溯原典段落、`graph.json` 和 `quality-report.md`。OCR 缓存存在时会直接复用，不重复转换 PDF。

灵签额外保留确定性结构字段：`number`/`number_label`（签号）、`level`（签级）和 `section`（签诗、圣意、解曰、释义、占验等原典栏目）。每支签生成一个 `qian:<book>:<number>` 节点，节点只通过 `contains` 边连接原典段落，不把不同栏目拼成未经来源支持的现代解读。

## 11. 一个例子（输入 → 产物）

**输入片段**（麻衣相法风格）：

```
卷二 · 掌纹篇
掌有三奇纹，主贵；明堂平满，财帛自聚。指纤长者聪俊，指短秃者愚贱。
```

**拆解产物**（一段 Passage）：

| 字段 | 值 |
|---|---|
| `chapter` | `卷二 · 掌纹篇` |
| `text` | `掌有三奇纹，主贵；明堂平满，财帛自聚。指纤长者聪俊，指短秃者愚贱。` |
| `topic` | `wealth`（命中「财帛」）|
| `tags` | `["明堂", "财帛", "掌纹"]`（术语表命中）|
| `plain` | `""`（未开 `--distill`；开则如「掌心开阔平满，是聚财的手相」）|
| `embedding` | `null`（当前关键词检索）|

问卦「我今年财运如何」时：主题 `wealth` 命中 `+2.0`，问题若含「财」标签再 `+1.2`，bigram 再叠加 → 该段被选中，作为引用卡片附在回答里：

> 「掌有三奇纹，主贵；明堂平满，财帛自聚。」——《麻衣相法 · 卷二 · 掌纹篇》

## 12. 设计要点回顾

- **整句进出**：永不破句，保证每段可独立引用、可溯源。
- **规则优先**：主题/标签用关键词规则，零成本、可解释、对古籍命中率高；LLM 只在「白话」这种确需生成的地方用。
- **wiki + 关键词起步，向量随时接**：`embedding` 列和检索抽象已铺好，pgvector 是「填数据 + 换打分」的一步升级，不重构。
- **版权干净**：只吃公有领域原文，白话由模型生成，不碰他人译注。
